# Philosophy

RunQuest is developed using an AI-first workflow.

AI is treated as part of the engineering team.

Every implementation begins with thinking.

Not coding.

---

---

# Rule 1

Documentation comes before implementation.

If a feature has no documentation,

it should not be implemented.

---

# Rule 2

Every implementation must reference:

Constitution

Vision

Product Principles

Game Design Pillars

MVP Definition

Technical Architecture

before coding begins.

---

# Rule 3

Small Pull Requests

Every task should be independently reviewable.

Prefer many small commits over one large commit.

---

# Rule 4

One Feature at a Time

Never develop multiple unrelated features simultaneously.

Complete one feature before starting another.

---

# Rule 5

Architecture Before Code

Every new system must answer:

Why does it exist?

Can it be simpler?

Can it be reused?

---

# Rule 6

AI Responsibilities

AI assists with:

Architecture

Game Design

Frontend

Testing

Documentation

Refactoring

Code Review

AI never changes product philosophy.

Only the Founder can change product direction.

---

# Rule 7

Definition of Ready

A task is ready when it contains:

Goal

Acceptance Criteria

Dependencies

Out of Scope

Technical Notes

---

# Rule 8

Definition of Done

A task is complete only if:

The feature works.

TypeScript has no errors.

No hardcoded user-facing text.

Supports English and Bahasa Indonesia.

Works after page refresh.

Uses LocalStorage correctly.

Matches Product Principles.

Matches Constitution.

---

# Rule 9

Refactoring

Refactor only when it improves:

Readability

Maintainability

Testability

Never refactor simply for preference.

---

# Rule 10

Measure Learning

Success is measured by:

What we learned.

Not by:

Lines of code.

Number of commits.

Number of features.

---

# Founder Workflow

Before every development session:

1. Read the Constitution.
2. Read the relevant documentation.
3. Review open tasks.
4. Confirm scope.
5. Implement the smallest valuable change.
6. Test.
7. Commit.
8. Update documentation if needed.

---

# AI Prompt Workflow

Every AI session should begin with:

Read all project documentation first.

Do not implement features that conflict with the Constitution.

Always prefer simpler solutions.

Explain trade-offs before implementation.

Challenge unnecessary complexity.

If requirements are ambiguous,

ask questions instead of making assumptions.

---

# Final Principle

Build products.

Not features.

Build habits.

Not sessions.

Build trust.

Not complexity.
